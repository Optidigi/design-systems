# Optidigi Design Systems

Multi-brand shadcn registry monorepo. One package per brand, all served from a single Docker container at `registries.optidigi.nl`.

## Layout

```
design-systems/
├── package.json              # monorepo root, pnpm workspaces
├── pnpm-workspace.yaml
├── Dockerfile                # builds all brands, serves via nginx
├── packages/
│   ├── siab/                 # SIAB design system (active)
│   │   ├── components.json
│   │   ├── registry.json
│   │   └── registry/default/
│   │       └── ... (.tsx source)
│   ├── insignia/             # placeholder for future
│   └── optidigi/             # placeholder for future
└── public/r/v1/              # built output (gitignored)
    ├── siab/
    │   ├── registry.json
    │   ├── button.json
    │   └── ... (one JSON per item)
    ├── insignia/
    └── optidigi/
```

Each `packages/<brand>/` is a self-contained registry: its own `components.json`, `registry.json`, and source files. They evolve independently — different themes, different customizations, different roadmaps.

## Build

```bash
pnpm install
pnpm build:all              # builds every brand → public/r/v1/<brand>/
pnpm build:siab             # builds only SIAB (faster iteration)
```

## Deploy (Docker)

The `Dockerfile` is multi-stage: build all packages with `pnpm build:all`, then serve `public/` via `nginx:alpine`. ~30MB final image, no runtime dependencies.

```bash
docker build -t optidigi-design-systems .
docker run -p 8080:80 optidigi-design-systems
# verify: curl http://localhost:8080/r/v1/siab/button.json | jq .name
```

In your VPS Docker host (Coolify / Dokploy / Traefik / whatever fronts containers), point `registries.optidigi.nl` at this container. HTTPS handled by your reverse proxy as for any other app.

## URL pattern

```
https://registries.optidigi.nl/r/v1/<brand>/<name>.json
```

Examples:
- `https://registries.optidigi.nl/r/v1/siab/button.json`
- `https://registries.optidigi.nl/r/v1/siab/theme.json`
- `https://registries.optidigi.nl/r/v1/insignia/button.json` (when insignia ships)

## Consuming from an app

In any consuming project's `components.json`:

```json
{
  "registries": {
    "@siab":     "https://registries.optidigi.nl/r/v1/siab/{name}.json",
    "@insignia": "https://registries.optidigi.nl/r/v1/insignia/{name}.json"
  }
}
```

Then:
```bash
# SIAB-branded apps
npx shadcn init --preset @siab/base
npx shadcn add @siab/button @siab/avatar

# Insignia-branded apps (when insignia is published)
npx shadcn init --preset @insignia/base
```

Apps can register multiple namespaces if they need to mix brands (rare).

## Adding a new brand

1. `cp -r packages/siab packages/<new-brand>`
2. Edit `packages/<new-brand>/registry.json`:
   - Top-level `name`: change to the new brand
   - All `registryDependencies: ["@siab/*"]` → `["@<new-brand>/*"]`
3. Edit `packages/<new-brand>/package.json`:
   - `name`: `@optidigi/<new-brand>-registry`
   - `registry:build` output path: `../../public/r/v1/<new-brand>`
4. Customize theme, components as needed
5. Add `build:<brand>` script to root `package.json` for convenience
6. `pnpm build:all` → verify `public/r/v1/<new-brand>/` is populated
7. Push → container rebuild → consumers can `shadcn add @<new-brand>/...`

## Versioning policy

- **Path-based, per-brand.** SIAB v1 lives at `r/v1/siab/`, SIAB v2 at `r/v2/siab/`. Brands version independently of each other.
- Within a major: additive changes only.
- Bump major: when breaking (renamed components, removed variants, large visual shift).

To cut SIAB v2:
1. Add `--output ../../public/r/v2/siab` script (alongside v1, both stay live)
2. Make breaking changes in `packages/siab/registry/default/`
3. Build, deploy. v1 remains untouched for consumers still on v1.

## Contribution flow

1. Edit a source file in `packages/<brand>/registry/default/<name>.tsx`
2. `pnpm build:<brand>` locally to sanity-check (output goes to `public/r/v1/<brand>/`, gitignored)
3. Commit the **source** changes only (`packages/<brand>/`). Built JSON is regenerated by Docker on deploy.
4. Push → Docker host rebuilds the image → consumers can `shadcn add @<brand>/<name> --overwrite`
5. CI in consuming apps (e.g. siab-payload) runs `registry:check` to fail on local drift

## What does NOT belong here

- App-level layout chrome bound to specific routing (`AppSidebar`, `SiteHeader`, `UserMenu`)
- Components bound to specific data models (`PagesTable`, `UsersTable`)
- Auth flows specific to one app
- Anything built only once — promote on the **third** occurrence (rule of three)
